AuDeRe: Automated Strategy Decision and Realization in Robot Planning and Control via LLMs[MIT 202511]

摘要

现有的大多数基于 LLM 的机器人研究，要么直接预测轨迹关键点（waypoints），要么在固定的工具集成框架内调用 LLM，从而限制了针对不同任务探索与配置最优解决方案的灵活性。

在本研究中，提出了一个框架，利用 LLM 根据任务描述、环境约束以及系统动力学特性，自动选择合适的规划与控制策略。随后，这些策略通过调用底层的通用规划与控制 API 得以执行。

实验对比简单的轨迹工作，带有时空约束的规划，迷宫导航以及。

使用prompt，模型自动选择规划以及控制策略，避免了手动调整以及专家知识。

介绍

模型并不直接生成轨迹或者code，LLM对任务需求、环境约束以及机器人动力学进行推理分析，并据此选择并调用相应的、功能完备的规划与控制应用编程接口（APIs），以实现最契合当前任务特性的控制与执行过程。

作为对比，LLM-predict以及LLM-code直接输出轨迹以及code。

  • 对比的评价指标：成功率，任务完成度，平均迭代query次数，基于迭代的成功率，不同的误差类别的详细分析。

Contribution：

  • 不直接生成轨迹或者代码，而是利用LLM选择合适的API进行调用，考虑环境约束以及任务细节等。
  • 不是将LLM融合到某一个算法框架single fixed tool，而是让LLM自主的根据内容，实现automated stategy selection。
  • 引入了迭代反馈机制，依据performance实现re-prompting。

任务描述：给定一个自然语言描述的任务，该任务包含了机器人动态，参考轨迹，初始状态，目标区域，环境约束，目标是LLM选择合适的规划和控制算法$A^\Tau\in A,其中A={A_1,A_2,…,A_N}$是一系列API。

remark:实际上LLM去调用API不是很简单的事情。要理解API参数以及实现的功能等。理清任务与API的关系。

API包含了一系列，从底层的轨迹跟踪，状态空间离散化到高层的规划都可以实现。总共设计了8个独立的API接口。并且通过模块化设计，具有很好的推广性以及通用性。

• A⋆ 搜索（astar）[10]：基于图的最优路径规划方法，通过启发式搜索高效地在离散状态空间和障碍物环境中导航。

• 交叉熵方法（CEM）[24]：系统性地探索工作空间，选择性能较优的样本，并通过迭代更新搜索分布逐步收敛至高质量解。

• 基于梯度的优化（grad）：利用 PyTorch 进行梯度下降与反向传播，实现可微分动力学与损失函数的轨迹优化，高效完成端到端可学习控制。

• 线性二次调节器（LQR）[2]：针对可线性化的动力系统执行最优跟踪控制，通过最小化状态与控制量的二次代价，实现精确的轨迹跟踪。

• 混合整数线性规划（MILP）：用于在高层时序逻辑（STL）约束下进行鲁棒的轨迹规划，可同时描述空间与时间约束。

• 模型预测控制（MPC）[8]：基于 CasADi 实现的预测优化控制方法，在有限预测时域内计算最优控制与规划解，支持线性与非线性动力学系统及其约束。

• PID 控制（PID）[31]：实现一种简单而有效的比例-积分-微分控制策略，适用于低复杂度的轨迹跟踪或航点跟随任务。

• *快速扩展随机树（RRT）[14], [13]：提供基于随机采样的路径规划方法，可采用标准 RRT 或优化版 RRT 算法，高效搜索*高维或障碍密集空间*中的可行路径。

一种 “从自然语言 → 算法选择 → 参数配置 → 代码执行” 的自动化流程。首先prompt让LLM选择某一个API，之后再去搜索API的详细信息，例如输入输出格式。进一步生成代码以及配置接口，生成可执行代码。

一个Round分为三个步骤：

  • 询问使用的API

  • 梳理问题的求解步骤以及后续的python实现

  • 如果代码失败了，round+1重复咨询。

实验

考虑五个规划和控制场景，难度逐渐提升。每个场景随机初始化，实验100次得到评价指标。跟踪过程中，mpc被调用，c场景调用了rrt，d场景调用A*以及rrt。所有实验都使用GPT-4o LLM进行推理。最大的迭代次数被限定在6次。

对比试验

  • 本文方法 LLM-use-API
  • end-to-end LLM-predict
  • LLM-code

评价指标：

  • 成功率
  • 成功使用的迭代次数
  • 随着迭代次数的成功率变化
  • 误差检测与分类
    • 解析错误（Parsing errors）
    • 语法错误（Syntax errors）
    • 超时错误（Timeout errors）
    • 任务失败（Task failures）

左侧是三个模型的成功率，右侧是平均查询次数，本文的方法还是占优势的。困难问题其他两个方法很难成功

相同的query次数，本文方法成功率高一些

三个模型不同的失败情况占比。本文方法大部分情况成功，task failure 也比较低。

消融实验

第一个是比较temperature的影响（温度影响模型输出的随机性）。温度低，输出结果更加deterministic，反之更加具有随机性。发现温度升高，成功率下降，并且query次数增加。

​ 另外还比较了不同的gpt系列对于成功率影响，但是没有很细致的结论。模型并不是越大越好，大模型成功率可能更低而且推理耗时。最后还比较了，只提供一个API信息和直接提供所有API信息的比较结果，发现只提供一个API效果反而更好一点。
